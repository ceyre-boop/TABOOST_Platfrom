🔐 TABOOST Firebase Security Guide

Goal: Secure platform where creators can ONLY access their own account, and admins can manage everything. No account hijacking possible.

📚 How Firebase Works (Simple)

1 Authentication (Who are you?)

Firebase Auth stores emails and passwords securely. When someone logs in, Firebase gives them a unique ID called a UID.

Example UID: XjD4aZVLhKVio5QTYEIcCbY0v0g2

This UID is their real identity. Email is just a label.

2 Firestore Database (Your data)

Stores user profiles, creator data, etc. Each user gets a document.

users/
  └── UID1/          ← Document ID = Firebase Auth UID
        email: "creator1@email.com"
        role: "creator"
        name: "Skyler"
  └── UID2/
        email: "marco@taboost.me"
        role: "admin"
        name: "Marco"
3 Security Rules (The Gatekeeper)

Rules decide who can read/write what. Example:

match /users/{userId} {
  allow read: if request.auth.uid == userId;
}

This means: "Users can ONLY read their own document."

🛡️ Why This Prevents Hijacking

When a creator logs in:

  1. Firebase verifies their password
  2. Firebase attaches their UID to every request
  3. If they try to access another user's data, the rule blocks it
✅ Result: Creators can ONLY see their own account. Impossible to access others.

4 Deploy Security Rules

In Cloud Shell, run:

cd ~/taboost-rules

cat > firestore.rules << 'EOF'
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Users can only access their own document
    match /users/{userId} {
      // Allow users to read their own data
      allow read: if request.auth != null && request.auth.uid == userId;
      
      // Allow users to update their own data (but not role)
      allow update: if request.auth != null 
        && request.auth.uid == userId
        && request.resource.data.role == resource.data.role; // Role cannot change
      
      // Only admins can create users
      allow create: if request.auth != null 
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'admin';
    }
    
    // Admin can read all users
    match /users/{userId} {
      allow read: if request.auth != null 
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'admin';
    }
    
    // Roster - approved creator list
    match /roster/{username} {
      // Anyone can read roster (to check if approved)
      allow read: if request.auth != null;
      
      // Only admin can modify
      allow write: if request.auth != null 
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'admin';
    }
    
    // Signup audit log
    match /signups/{docId} {
      allow create: if request.auth != null;
      allow read: if request.auth != null 
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'admin';
    }
  }
}
EOF

🚨 Common Mistakes to Avoid

❌ NEVER DO THIS: 
// BAD - Lets anyone read everything
match /{document=**} {
  allow read, write: if request.auth != null;
}
❌ NEVER USE EMAIL AS DOCUMENT ID: 
// BAD
users/
  └── marco@taboost.me  ← Don't do this

// GOOD
users/
  └── XjD4aZVLhKVio5QTYEIcCbY0v0g2  ← Use UID
